# FairinoCutTest: движение робота по точкам из файла

## Почему у вас ошибка MSB3644
Ошибка

`MSB3644: не найдены ссылочные сборки для .NETFramework,Version=v4.8.1`

означает, что на ПК **не установлен .NET Framework 4.8.1 Developer Pack (Targeting Pack)**.
Проект `FairinoCutTest` и SDK `FRRobot` собираются под `net481`, поэтому без targeting pack сборка невозможна.

## 1) Что обязательно установить на Windows
Установите **один** из вариантов:

1. **Visual Studio 2022** с компонентом:
   - `.NET Framework 4.8.1 SDK`
   - `.NET Framework 4.8.1 targeting pack`
2. Или **Build Tools for Visual Studio 2022** с теми же компонентами.
3. Или напрямую **.NET Framework 4.8.1 Developer Pack**:
   - https://aka.ms/msbuild/developerpacks

После установки обязательно **перезапустите PowerShell/терминал**.

## 2) Проверка, что targeting pack установлен
В PowerShell:

```powershell
Test-Path "C:\Program Files (x86)\Reference Assemblies\Microsoft\Framework\.NETFramework\v4.8.1"
```

Если команда вернула `True` — можно собирать проект.

## 3) Запуск симулятора и приложения
### 3.1 Поднимите контейнер симулятора
(как в вашем примере с `-p ...` и `--net fairino-net`).

### 3.2 Запустите приложение
Из каталога `FairinoCutTest`:

```powershell
dotnet restore
dotnet run -- 127.0.0.1 trajectory_infinity.json
```

> Если приложение запускается в другом контейнере, используйте IP/hostname из `fairino-net`, а не `127.0.0.1`.

## 4) Формат файла точек
Файл по умолчанию: `trajectory_infinity.json`.

Параметры:
- `tool`, `user` — номера инструмента/системы координат.
- `velocity`, `acceleration`, `ovl`, `blendRadius` — параметры движения.
- `waitMotionDone` — ждать завершения каждой точки.
- `startPointWithMoveJEachLoop` — входить в первую точку цикла через `MoveJ` (`true`) или идти непрерывно `MoveL` (`false`).
- `motionDonePollMs` — период опроса завершения движения.
- `queuePollMs` — период опроса очереди робота, когда программа ограничивает предварительную постановку команд в очередь.
- `maxQueuedMotionSegments` — максимум заранее поставленных сегментов при `waitMotionDone: false`; по умолчанию `20`. Это не даёт программе мгновенно закинуть сотни циклов в контроллер, поэтому `Ctrl+C` остаётся управляемой остановкой, а очередь не забивается до `255`. Значение `0` отключает ограничитель.
- `loopCount` — количество циклов: `0` = бесконечно. Даже при конечном `loopCount` и `waitMotionDone: false` программа после постановки команд остаётся запущенной и ждёт опустошения очереди, чтобы `Ctrl+C` мог остановить уже поставленное движение.
- `points` — список поз `{x,y,z,rx,ry,rz}`.


## 4.1 Как сделать движение более плавным (без остановок на точках)
Для непрерывной траектории обычно помогают такие настройки:
- `waitMotionDone: false` — не ждать полного завершения движения в каждой точке;
- увеличить `blendRadius` (например, `10..30`) — больше скругление и меньше «торможение» в вершинах;
- `startPointWithMoveJEachLoop: false` — не делать жёсткий `MoveJ` в начале каждого цикла.

Пример:
```json
{
  "blendRadius": 20,
  "waitMotionDone": false,
  "maxQueuedMotionSegments": 20,
  "queuePollMs": 30,
  "startPointWithMoveJEachLoop": false
}
```

> Важно: слишком большой `blendRadius` может заметно увести траекторию от исходных точек, подбирайте по месту.


## 4.2 Диагностика IK, LIN-пути и обход через промежуточную высоту

Программа перед движением теперь печатает текущую TCP-позу и суставы робота. Это удобно сверять с окном симулятора: если в симуляторе `Calculate joints` даёт рабочую ориентацию `RX/RY/RZ`, используйте такие же углы в JSON или включите `useCurrentTcpOrientation`.

Дополнительные параметры JSON:
- `useCurrentTcpOrientation` — заменить `rx/ry/rz` всех точек на текущую ориентацию TCP робота при запуске. Полезно, если в симуляторе рабочая ориентация сильно отличается от `180,0,0`.
- `pathCheckStepMm` — шаг дискретной проверки прямого `MoveL`-сегмента. Программа проверяет не только конечные точки, а промежуточные точки линии.
- `maxJointStepDeg` — максимально допустимый скачок любого сустава между соседними проверочными точками. Проверяются все оси J1..J6; если хотя бы одна ось превышает лимит, прямой `MoveL` считается опасным: возможны лимит, сингулярность или смена конфигурации.
- `maxJointStepDegByAxis` — необязательные отдельные лимиты для J1..J6, например `[45,45,45,30,30,45]`. Если значение для оси не задано или `<= 0`, используется общий `maxJointStepDeg`.
- `autoViaZ` — если прямой `MoveL` не проходит проверку, программа пробует построить альтернативный путь: подняться на `autoViaZ`, пройти горизонтально и опуститься в целевую точку. Если и этот путь не проходит IK/path-check, движение не стартует.
- Для первой точки с `startPointWithMoveJEachLoop: true` программа сначала пробует IK относительно текущих суставов, а если это не получилось — свободный IK (`GetInverseKin` с конфигурацией `-1`). Это позволяет `MoveJ` перейти в другую допустимую конфигурацию, когда текущая поза робота сильно отличается от стартовой точки траектории.

Пример осторожного запуска рядом с текущей ориентацией TCP:

```json
{
  "velocity": 10,
  "acceleration": 10,
  "ovl": 30,
  "blendRadius": 0,
  "waitMotionDone": true,
  "loopCount": 1,
  "startPointWithMoveJEachLoop": true,
  "precheckPathWithIk": true,
  "useCurrentTcpOrientation": true,
  "pathCheckStepMm": 20,
  "maxJointStepDeg": 30,
  "autoViaZ": 160
}
```

Важно: `autoViaZ` — не полноценный 3D-планировщик коллизий. Он помогает обойти проблемный прямой LIN-сегмент, но реальные столкновения всё равно контролируются realtime-состоянием робота (`collisionState`, safety stop, `main_code/sub_code`) во время движения.

## 5) Частые проблемы
- `RPC connect failed`:
  - неверный IP;
  - контейнер не поднят;
  - приложение не видит сеть/порт.
- Робот не двигается при успешном подключении:
  - проверь режимы симулятора;
  - проверь корректность `tool/user` и достижимость точек.

- Ошибка `err=14` / сообщение про изменение joint configuration или singular pose:
  - в первую точку лучше входить через `MoveJ`, а не сразу `MoveL`;
  - для каждой Cartesian-точки нужно считать IK относительно текущей конфигурации суставов.

- Ошибка `err=112` (ERR_TARGET_POSE_CANNOT_REACHED):
  - целевая поза недостижима для текущей конфигурации (IK не находит решение);
  - причиной может быть выход за рабочую область, неудобная ориентация TCP, близость к сингулярности, ограничение суставов или коллизионное/защитное состояние.
- Если срабатывает столкновение/защита:
  - проверяйте realtime-state (`collisionState`, `EmergencyStop`, `main_code/sub_code`);
  - уменьшайте траекторию (x/y), поднимайте z, меняйте ориентацию `rx/ry/rz`;
  - включайте предварительную IK-проверку траектории до запуска цикла.
